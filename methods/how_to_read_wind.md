## 1. 「風を読む」とはどういう作業か

### 定義

風を読むとは、**観測できる表面への作用から、観測できない裏面の変形を予想する**作業である。

これは以下の三点において、通常の「分析」とは異なる。

```
通常の分析：
  観測したデータから現状を記述する
  → 「今どうなっているか」を正確に把握する

風を読む：
  観測したデータから見えないものを推定し
  まだ起きていないことを予想する
  → 「次に何が起きるか」を先取りする
```

### 技術としての「風を読む」

気象予報士は「天気を感じる」だけでなく、手順を持っている。

```
気象予報士の手順：
  観測データを収集する
    ↓
  数値モデルを走らせる
    ↓
  モデルの出力を「経験と直感」で補正する
    ↓
  複数のシナリオを並置する
    ↓
  最も確率の高い予報を選ぶ
    ↓
  実際の天気と照合し、予報精度を評価する
```

OOπ理論の七つのステップは、この気象予報士の手順を**あらゆる領域に適用できる形**に一般化したものである。

### 何が「技術」で何が「才能」か

```
才能（生まれつきのもの）：
  異変への感度
  パターン認識の速さ
  複数の仮説を同時に保持する能力

技術（学べるもの）：
  観測の手順
  変数の特定方法
  フライトプランの設計
  照合と更新の方法
  乖離の分類
```

このドキュメントは技術の部分を記述する。才能の部分は経験によって培われる。

## 2. 七つのステップの全体像

```
STEP 1：観測（表面を見る）
  何が観測できるか。死角はどこか。

    ↓

STEP 2：感知（異変を感じる）
  「何かおかしい」という直感を言語化する。

    ↓

STEP 3：変数の特定（何が動いているか）
  TTT式のどの変数が支配的か。
  第7項以降の変数の有無を確認する。

    ↓

STEP 4：iIの推定（見えないものを描く）
  潜在的に蓄積しているものは何か。
  E式のiI項を推定する。

    ↓

STEP 5：フライトプランの設計（予想を三本立てにする）
  主シナリオ・代替シナリオ・外れ値シナリオ。

    ↓

STEP 6：照合（実際と比べる）
  実際に何が起きたか。
  どのシナリオに最も近かったか。

    ↓

STEP 7：理論の更新（外れから学ぶ）
  乖離の種類を特定し、欠落変数を発見する。
  次の観測に反映する。
```

このサイクルは一回で終わらない。繰り返すことで精度が上がる。

## 3. ステップ1：観測（表面を見る）

### 目的

「今、観測できているものを正確に把握する」

### 手順

**① 観測できているものをリストアップする**

```
問い：何が見えているか？
  定量的なもの（数値・割合・頻度）
  定性的なもの（言葉・行動・関係）
  時間的なもの（変化・傾向・速度）
  空間的なもの（場所・分布・境界）
```

**② 観測できていないものを特定する**

```
問い：何が見えていないか？
  測定されていないが存在するもの
  測定されているが自分が見ていないもの
  原理的に観測できないもの
  観測しようとすると変化してしまうもの
```

**③ 観測の「品質」を評価する**

```
問い：観測データはどの程度信頼できるか？
  データの鮮度（いつ時点のか）
  データの代表性（何を代表しているか）
  観測者のバイアス（誰が何のために集めたか）
  欠損データの存在（ないことに意味があるか）
```

### 気象での適用例

```
観測できているもの：
  気圧（数値）・気温（数値）・湿度（数値）
  衛星画像（雲の分布・形状）
  レーダー（降水域・強度）
  各地のアメダスデータ

観測できていないもの：
  上空のジェット気流の詳細な構造
  海上の水蒸気量（観測点が少ない）
  地盤の状態（能登の地震後の脆弱化）

観測の品質：
  気圧・気温：高品質（自動観測・定期更新）
  雨量：中品質（観測点の間隔に依存）
  地盤状態：低品質（地震後の変化が未反映）
```

### 社会・組織での適用例

```
観測できているもの：
  世論調査の数値・選挙結果・経済指標
  報道・公式発表・公開情報
  組織のKPI・業績数値

観測できていないもの：
  組織内部の会話・個人の動機
  報道されない出来事・非公開情報
  数値に現れない疲弊・不満

観測の品質：
  世論調査：バイアスに注意（回答者の選択性）
  公式発表：意図的な情報操作の可能性
  KPI：「測定されやすいもの」しか測定しない
```

## 4. ステップ2：感知（異変を感じる）

### 目的

「データになる前の『気配』を捉える」

### 感知とは何か

感知はステップ1の観測より前に起きる場合がある。「何かおかしい」という直感が、観測の方向を決めることがある。

しかし感知は誤りやすい。個人的なバイアス・過去の経験への過度な依存が感知を歪める。

**感知の役割：観測の「方向」を示すこと。観測に代わることではない。**

### 感知のシグナル

```
通常とは異なる「静けさ」：
  動くべきものが動いていない
  例：「いつもうるさい人が急に静かになった」
     「気圧がいつもと同じなのに空気が違う感じがする」

通常とは異なる「騒がしさ」：
  動かないはずのものが動いている
  例：「平静を装っているが微妙に挙動が変わった」
     「数字は変わっていないが現場の様子が違う」

複数の観測点での「ずれ」：
  一点では説明できない不一致
  例：「気圧は正常だが温度が異常に高い」
     「業績は良いが従業員の表情が暗い」

「前歴（hH）」との断絶：
  過去のパターンが突然通じなくなった感覚
  例：「いつも同じ手が効かなくなった」
     「去年と同じ条件なのに今年は反応が違う」
```

### 感知を言語化する練習

感知は言語化しにくい。しかし言語化することで：
- 他者と共有できる
- 観測の方向が具体化する
- 後から「感知が正しかったか」を検証できる

**言語化のテンプレート：**

```
「今、[何か]が[通常と違う]という感覚がある。
 具体的には[観測できる手がかり]がその根拠だが、
 うまく言語化できない部分もある。
 この感覚が正しければ、[何か]が起きているはずだ。」
```

## 5. ステップ3：変数の特定（何が動いているか）

### 目的

「TTT式のどの変数が、今この状況を支配しているかを特定する」

### 手順

**① 基本六変数の有無と方向を確認する**

```
チェックリスト：

xX（水平方向の力・対立軸）
  → この系に水平方向の力・勾配・対立はあるか？
  → ある場合：どの方向に・どの強さで？

yY（縦方向の力・権威構造）
  → 上下方向の権威・温度差・ヒエラルキーはあるか？
  → ある場合：どの程度集中しているか？

zZ（鉛直方向の力・チェック機能）
  → 変換・チェック機能は実際に働いているか？
  → ない場合：なぜ機能していないか？

Uu（時間変化率）
  → 状態は変化中か、停滞しているか？
  → 変化率は加速中か・減速中か・ゼロか？

vV（流体・意思・フラックス）
  → 意思・エネルギー・情報の流れはあるか？
  → どの方向に・どの速さで流れているか？

wW（エネルギー移流・正統性）
  → エネルギー・正統性はどこに集中しているか？
  → その集中は安定しているか・変化しているか？
```

**② 支配的な変数を特定する**

```
問い：今この状況を最も説明する変数はどれか？

判断基準：
  その変数の係数を変えると、
  予想が最も大きく変わる変数が支配的。

例（低気圧の形成）：
  xX（気圧勾配）は背景場として機能
  yY（南北温度勾配）で前線位置が決まる
  zZ（上昇気流）で降水が決まる
  vV（水蒸気）で降水量が決まる ← 支配的
  → 「vVがこの状況を支配している」

例（組織の崩壊）：
  xX（部門間対立）は表面的
  yY（権威の集中）が根本原因
  zZ（内部監査の機能不全）が問題を見えにくくしている
  Uu（変化率の急上昇）が引き金
  → 「yZの相関（行政がチェックを支配）とUuの急変が支配的」
```

**③ 第7項以降の有無を確認する**

```
六変数で説明できない残差があれば：

hH（前歴・履歴）：
  → この系には、現在の状態を規定する過去の変形があるか？

sS（スケール・規模）：
  → 分析しているスケールは適切か？
  → 局地現象を広域の変数で説明しようとしていないか？

eE（境界条件・外部環境）：
  → 系の外から課されている制約・強制入力はあるか？
  → 外部からの力が内部変数を上書きしていないか？

nN（非線形臨界）：
  → 閾値を超えようとしている要素はあるか？
  → 急激な質的変化の前兆はあるか？

cC（連鎖・カスケード）：
  → 隣接する系からの影響を受けていないか？
  → この系の変化が他の系に波及していないか？

dD（散逸・抵抗）：
  → 変化のエネルギーを消耗させている力はあるか？
  → なぜ変わるべきものが変わらないか？
```

## 6. ステップ4：iIの推定（見えないものを描く）

### 目的

「E式のiI（潜在量）を推定する：まだ表面に現れていないが蓄積しているものを描く」

### iIを推定するための問い

```
問い1：「なぜ予想より大きな変形が起きたか」の説明として
       何があれば説明できるか？

問い2：「表面のデータからは見えないが、
       確かに存在するはずのもの」は何か？

問い3：「過去の乖離パターンから見て、
       今回も欠落しそうな変数は何か？」

問い4：「この系の前歴（hH）から見て、
       どんなiIが蓄積しているか？」
```

### iIの推定方法

**間接的シグナルからの推定：**

```
シグナル①：変化率Uuの急加速
  → iIが臨界（nN）に近づいているサイン
  → 表面には見えないが内部で何かが進行している

シグナル②：複数の観測点での同期した変化
  → iIが系全体に広がっているサイン
  → 局所的な問題ではなく構造的な問題

シグナル③：「一貫して外れる方向」の乖離
  → iIが特定の方向に働いているサイン
  → 欠落変数が一定の方向性を持っている

シグナル④：既存モデルの精度低下
  → iIが既存の理解では捉えられない何かであるサイン
  → 新変数の存在を示唆
```

**具体例：能登大雨でのiI推定**

```
観測データ：
  気象変数（vV・wW・zZ）は正確に把握できていた
  「大雨になる」という予報は当たっていた

乖離：
  「被害規模」が予想を大幅に超えた

iIの推定：
  「地震後の地盤脆弱化」が存在するはずだ
  → これを考慮すれば、同じ降水量でより大きな被害が説明できる

検証：
  hH（地盤前歴）という変数として定式化
  P_応答 = P_入力 × f(hH)
  f > 1（脆弱化）で説明できることを確認
```

### iI推定のチェックリスト

```
□ 観測できていないが存在が推測されるものをリストアップした
□ そのiIがあれば乖離が説明できることを確認した
□ そのiIを示す間接的シグナルを特定した
□ 複数の独立した事例で同じiIが欠落していないか確認した
□ iI→rR転換の条件（nN超過）を推定した
```

## 7. ステップ5：フライトプランの設計（予想を三本立てにする）

### 目的

「一つの予想に固執せず、複数のシナリオを並置して、予想の経路を設計する」

### フライトプランの三本立て

```
主シナリオ：最も可能性が高い裏面の変形
  根拠：支配的な変数の方向
  確率の感覚：高
  判断基準：このシナリオが外れた時の「シグナル」を設定する

代替シナリオ：主シナリオが外れた場合の次善の予想
  根拠：第二・第三の支配変数
  確率の感覚：中
  判断基準：どの観測値が出たら代替シナリオに切り替えるか

外れ値シナリオ：低確率だが起きた場合の影響が大きいもの
  根拠：nN超過・eE急変・cCの連鎖
  確率の感覚：低
  判断基準：どの初期シグナルが見えたら注目するか
```

### フライトプランの書き方

```
【主シナリオ】
  予想する裏面の変形：
  根拠となる変数と係数：
  転換の条件（iI→rR）：
  タイムライン（いつ頃起きるか）：
  「外れた」と判断するシグナル：

【代替シナリオ】
  予想する裏面の変形：
  主シナリオとの違い：
  切り替えのトリガー：

【外れ値シナリオ】
  最悪ケース / 最善ケース：
  その場合の変数の状態：
  監視すべき早期シグナル：
```

### フライトプランの注意点

```
注意①：単一の予想に固執しない
  「絶対これだ」と思った瞬間、
  外れた時の気づきが遅れる

注意②：確率を数字にしすぎない
  「75%の確率で」という精度は存在しない
  「高・中・低」の三段階で十分

注意③：「外れたら気づけるか」を事前に設計する
  どの観測値が変化したら予想を修正するか
  を事前に決めておく

注意④：「外れた場合の行動」を事前に考える
  予想が外れた時のフォールバックを持つ
  これがフライトプランが「計画」である理由
```

## 8. ステップ6：照合（実際と比べる）

### 目的

「実際に何が起きたかを、フライトプランのどのシナリオと照合する」

### 三列比較の実施

```
観測地点 | TTT理論予報（自分の予想） | データ予報（既存モデル） | 実際の現象
--------|-------------------------|----------------------|--------
地点A   | 主シナリオの予想          | 既存の分析・常識的予想  | 実際の結果
地点B   | 同上                     | 同上                  | 同上
```

### 照合の判定基準

```
「一致」：実際の現象が主シナリオの予想範囲内に収まった

「部分一致」：
  主シナリオが当たったが規模・タイミングが違った
  代替シナリオの方が近かった

「乖離」：
  どのシナリオとも一致しなかった
  予想の方向性は合っていたが規模が大きく違った

「質的乖離」：
  現象の「種類」が違った（最も重要な乖離）
  → 式に入っていない変数が存在するサイン
```

### 照合で重要なこと

```
① 「なんとなく当たった」は照合ではない
  どのシナリオのどの予想が当たったかを具体的に確認する

② 「なんとなく外れた」は照合ではない
  乖離の種類（量的・位置的・時間的・質的）を特定する

③ 「外れた時ほど丁寧に記録する」
  当たった記録より外れた記録の方が理論を育てる

④ 「既存モデルとの差」に注目する
  TTT理論が既存の分析より当たった点・外れた点を特定する
  その差が理論の「強み」と「改善点」を示す
```

## 9. ステップ7：理論の更新（外れから学ぶ）

### 目的

「乖離を分析し、欠落していた変数を発見し、次の観測に反映する」

### 乖離の四分類

```
量的乖離（大きさが違う）：
  → どの変数の係数が誤っていたか？
  → 係数の修正で対応できるか？
  → それとも別の変数が係数を変えていたか（hHの効果）？

位置的乖離（場所が違う）：
  → 空間的な解像度が不足していたか？
  → sS（スケール）の設定が適切でなかったか？
  → 地形・地理的な要因（eE）を見落としていたか？

時間的乖離（タイミングが違う）：
  → Uu（変化率）の読み誤りか？
  → nN（臨界）の到達タイミングの予測精度の問題か？
  → cC（連鎖）のタイムラグを考慮していなかったか？

質的乖離（現象の種類が違う）：
  → 最も重要。P式に入っていない変数が存在するサイン。
  → どんな変数を追加すれば説明できるか？
  → その変数は既存変数と独立しているか？
```

### 欠落変数の発見プロセス

```
Step 1：乖離を記述する
  「Xが起きると予想したが、Yが起きた」

Step 2：「Yが起きるために必要な条件」を列挙する
  「YがXではなくYになるためには何が必要だったか」

Step 3：その条件が「観測できていなかった変数」に対応するか確認する
  「その条件を記述する変数はP式の中にあるか」

Step 4：なければ新変数の候補として記述する
  「どの文字・どんな定義で記述するか」

Step 5：他の事例での検証
  「同じ変数が他の事例でも欠落していないか」
```

### 能登大雨での理論更新の例

```
乖離の記述：
  「大雨は予報できたが、被害規模が予想を超えた」
  （質的乖離：「雨が降る」は当たったが「被害の大きさ」が外れた）

Yが起きるために必要な条件：
  「地震後の地盤脆弱化」があれば説明できる

新変数の候補：
  hH（前歴・履歴）：系の過去の変形が現在の応答を変える
  P_応答 = P_入力 × f(hH)

他の事例での検証：
  司法機能不全：砂川事件というhHが現在のzZを規定している
  社会OS：hHの第二層が第一層の変更を無効化する
  → 複数の独立した領域で確認 → 確定変数として採用
```

### 更新した理論を次の観測に反映する

```
記録の更新：
  log/template.md の「理論への示唆」欄に記録する
  新変数の候補は theory/04_variables.md に追記する

次の観測への反映：
  次回のSTEP 3（変数の特定）で
  新変数の有無も確認リストに加える

仮説の精緻化：
  「hHがある場合」「ない場合」で
  フライトプランを分岐させる
```

## 11. よくある失敗と対策

### 失敗①：観測と感知を混同する

```
失敗：
  「なんとなくそんな気がした」を
  「データが示している」と表現する

対策：
  観測（数値・事実）と感知（直感・解釈）を
  明示的に区別して記録する
  「これは観測か、感知か」を常に確認する
```

### 失敗②：支配的な変数を一つに絞れない

```
失敗：
  「全部重要」と言って
  実質的にフライトプランが作れない

対策：
  「係数を変えると予想が最も大きく変わる変数」
  という基準で強制的に一つを選ぶ
  その後、第二・第三の変数を補足する
```

### 失敗③：フライトプランが一本しかない

```
失敗：
  「必ずこうなる」と確信して
  代替シナリオを持たない

対策：
  「この予想が外れる条件は何か」
  を必ず考える
  外れる条件が思いつかない時は、
  自分がバイアスにはまっている可能性がある
```

### 失敗④：照合を「当たったかどうか」だけで評価する

```
失敗：
  「当たった → 理論が正しい」
  「外れた → 理論が悪い」
  で終わる

対策：
  「なぜ当たったか」「なぜ外れたか」を
  必ず変数レベルで分析する
  当たった理由が間違っていることもある
```

### 失敗⑤：外れた記録を残さない

```
失敗：
  当たった記録だけを記録し
  外れた記録を見ないようにする

対策：
  外れた記録を最も丁寧に記録する
  「外れた記録こそが理論を育てる」
  という認識を持つ
```

### 失敗⑥：第7項以降を後付けで説明に使う

```
失敗：
  外れた後で「hHがあったから」と
  後付けで説明する

対策：
  STEP 3で第7項の有無を「事前に」確認する
  後付けの説明は「次回の予想への反映」として使う
  「後から見ればわかった」を減らすのが目的
```
